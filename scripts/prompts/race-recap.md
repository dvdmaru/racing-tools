# 賽後戰報寫手 prompt（type: report）

給主寫子代理（Sonnet）的完整指令。**這份檔案是硬規則的唯一載體**——規則寫在這裡，
不是靠人記得在對話裡複述。漏講一條，稿子就會少一道防線。

呼叫方式（Claude Code Agent tool，非 headless CLI）：

```
Agent(subagent_type="claude", model="sonnet", prompt=<本檔內容> + facts 路徑 + slug)
```

---

## 你的任務

依照下面的 facts pack 寫一篇賽後戰報，輸出到 `articles/<slug>/index.md`。

**facts pack 路徑**：`facts/race-recap-<season>-r<NN>.json`（呼叫時會給你確切路徑）

---

## 鐵則（違反任何一條，稿子會被退回重寫）

0. **命中即改稿，沒有豁免。** 發布前的機械對帳有四道，任何一道紅燈都必須靠改稿
   解決——沒有「寫進豁免清單放行」這個選項。數字對不上就拿掉那句話，
   因果句被抓到就改寫。（豁免若由產稿者自己填，等於作者能把自己的紅燈改綠。）

1. **禁 free recall。** 文中每一個數字、每一個人名、每一個車隊名，都必須在 facts pack
   裡找得到。你對這個賽季的既有印象一律不採信——賽果會因判罰更動，你的記憶不會。
   需要某個數字但 pack 裡沒有 → 不要寫那句話，不要估算，不要用「約」。

2. **只寫「發生了什麼」，不寫「為什麼」。** 這是戰報不是分析。
   - ✅「羅素從第 3 位起跑，第 0 圈退賽。」
   - ❌「羅素因為起步失誤被擠出賽道。」——因果歸屬屬於賽後分析文型。
   - 車隊的內部意圖（決定、選擇、放棄、賭一把）一律不可斷言。

3. **未完賽的寫法。** 用 facts pack 的 `finish_state` 欄位，不要自己判斷：
   `完賽` / `完賽（遭套圈）` / `未完賽`。
   `status` 是英文原文（`Retired`、`Accident`、`Engine`），**照掛不要直譯成因果句**。
   ⚠️ 這個欄位是**資料源（jolpica，志願者維護的開源專案）的狀態標示，不是 FIA 官方分類**——
   不可寫成「官方認定」「官方分類為」。「Engine」只是資料源的一個標籤，
   不等於你可以寫「引擎爆缸讓他失去一切」。
   ⚠️ 特別注意：status 是 `Lapped` 的車手**有完賽名次**，寫成「退賽」是事實錯誤。

4. **紀錄型主張要有支撐。** 「本季最大逆轉」「隊史首次」「史上第一」這類句子，
   只有 `record_claims` 欄位裡有對應項目才能寫。那個欄位是空的 → 一律降成畫面描述
   （「從第 21 位起跑拿到第 6」比「本季最大逆轉」既安全又具體）。

5. **中英對照（站規）。** 車手／車隊／賽道／站名第一次出現時中英並列，
   中文在前、原文在後，中間用全形空格。之後可只用中文。
   用 pack 裡的 `zh` 與 `en_full`／`en`，不要自己翻譯。

6. **IP 紅線。** 站名不出現 F1 字樣；不描述、不引用任何官方素材（會徽、隊徽、官方照片、
   官方轉播畫面）；不寫「官方認證」之類暗示關聯的措辭。

7. **繁體中文 ＋ 全形標點。** 不出現簡體字。逗號句號一律全形，數字用阿拉伯數字，
   中英文之間加盤古之白。不要用港式詞彙。

8. **去 AI 味。** 不用「不僅…更…」「值得注意的是」「展現了…的決心」這類套語；
   不要每段都同樣長度；結尾不要 CTA、不要「讓我們拭目以待」。

## 結構（1,800-2,500 字，含標點）

1. **導言**（2-3 句）：一句話講結果，再點出一個轉折。不要鋪陳。
   ⚠️ **轉折只能取自 `timeline` 欄位**（`lead_changes`／`laps_led_by`／
   `unexplained_position_gains`／`pit_strategy`）。`timeline` 是 `null` 就代表逐圈資料
   還沒落地——這時**不要寫轉折**，導言只寫結果。沒有資料就少寫一句，不要補。
2. **前十完賽表**：markdown 表格，欄位＝名次／車手／車隊／發車位／積分。
   數字全部取自 `top10`。
3. **關鍵段落 3-4 段**，素材依序取自：
   - `timeline.lead_changes` / `laps_led_by`：誰領先了幾圈、第幾圈易主
   - `timeline.unexplained_position_gains`：未解釋的名次上升
   - `timeline.pit_strategy`：幾停、第幾圈進站、停留秒數
   - `biggest_gainers` / `biggest_losers`：發車位與完賽名次的最大落差
   - `dnf`：未完賽名單與資料源狀態標示
   - `fastest_lap`：最快單圈

   ⚠️ **`unexplained_position_gains` 不等於「超車」。** 欄位名就是它的定義：
   已排除「自己進站」「被超過的人進站」「該圈有人退場遞補」三種情形之後，
   **仍然沒有解釋**的名次上升。沒有解釋不代表就是超車——逐圈名次無法排除打滑、
   讓位、罰則。所以只能寫成「第 N 圈上升到第 X 位」這種位置描述，
   **不可寫「超越了誰」**，你不知道對方當時發生什麼事。
   ⚠️ **絕對不可寫的三件事**：輪胎配方與輪胎策略、安全車、事故經過。
   這個資料源完全沒有這三類資料，寫出來一定是編的。「因為換上軟胎所以追上」
   這種句子即使結論碰巧對，也是零根據。
4. **積分榜變化**：用 `standings.drivers_before` vs `drivers_after`（車隊榜同理），
   講前五的位次與差距變化。`leader_change` 為 false 就不要寫「易主」。
5. **下一站預告**：用 `next_race`，附台北時間，一句話帶過。

## 你不需要做的事

- 不用寫 FAQ（本文型不掛 FAQPage schema）
- 不用配圖、不用寫封面（封面由既有數據視覺流程處理）
- 不用 build、不用部署——**寫完就停**，後面是對帳與人工審稿
